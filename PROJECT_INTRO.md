# 河口医院智能协同办公平台

## 一、项目概述

本项目是面向医院行政管理场景设计开发的智能协同办公（OA）系统，旨在解决传统医院行政办公中审批流转效率低、文档管理分散、跨科室协同困难、患者信息查阅不便等痛点。系统采用 **Spring Cloud 微服务架构**，前后端分离设计，涵盖审批中心、通知文档、会议任务、通讯录、患者服务、系统管理等功能模块，为医院日常行政办公提供一站式数字化解决方案。

**项目名称**：河口医院智能协同办公平台（Hospital OA System）

**技术定位**：基于 RuoYi-Cloud 微服务框架的二次开发，在保留其成熟的权限管理、代码生成等基础能力的同时，针对医院 OA 场景自主设计并实现了全部业务模块。

## 四、功能模块详述

### 4.1 工作台

系统首页仪表盘，聚合展示当前用户的待办事项、近期日程、通知公告等关键信息，帮助用户快速掌握工作全局。

### 4.2 审批中心

| 功能 | 说明 |
|------|------|
| 我的申请 | 发起请假、报销、用印、物资等各类 OA 申请单，支持查看申请进度 |
| 我的待办 | 查看待审批的任务列表，进行审批操作（通过/驳回） |
| 我的已办 | 查看已处理的审批记录，支持审批历史追溯 |

**核心表**：`oa_workflow_apply`（申请单）、`oa_workflow_task`（审批任务）

### 4.3 通知文档

| 功能 | 说明 |
|------|------|
| 通知中心 | 系统公告和通知的发布与管理 |
| 制度文档 | 医院制度文件的分类管理，支持文档的新增、编辑、发布、归档全生命周期管理，附件上传支持 MinIO 对象存储 |

**核心表**：`sys_notice`（公告通知）、`oa_document`（文档）、`oa_document_category`（文档分类）、`oa_document_attachment`（附件）、`oa_document_read_log`（阅读记录）

### 4.4 会议任务

| 功能 | 说明 |
|------|------|
| 会议管理 | 会议室资源管理、会议预约与安排、会议纪要记录 |
| 任务督办 | 任务发布与指派、任务进度跟踪、跟进记录管理 |
| 日程提醒 | 个人日程安排与提醒，日历视图展示 |

**核心表**：`oa_meeting_room`（会议室）、`oa_meeting`（会议）、`oa_task`（任务）、`oa_task_record`（任务跟进记录）、`oa_schedule`（日程）

### 4.5 通讯录

| 功能 | 说明 |
|------|------|
| 通讯录查询 | 按科室、姓名快速检索院内人员联系方式，支持联系人分组 |
| 人员管理 | 员工账号、岗位、档案信息的增删改查及批量导入导出 |
| 科室管理 | 医院组织架构的树形管理，支持多层级科室体系 |

**核心表**：`oa_staff_profile`（人员档案扩展）、`oa_department_ext`（科室扩展）、`oa_contact_group`（联系人分组）、`sys_user`（用户基础信息）、`sys_dept`（科室基础信息）

### 4.6 患者服务

| 功能 | 说明 |
|------|------|
| 患者工作台 | 患者总数、在院患者、活动病例、已结案病例等统计概览 |
| 患者查询 | 按姓名、编号、住院号、身份证末四位等多维检索患者，支持按科室、状态（在院/留观/已出院）筛选 |

| 访问审计 | 记录每次患者信息查看行为，含操作人、科室、IP 地址等，保障患者隐私安全 |

**核心表**：`oa_patient`（患者主档）、`oa_patient_case`（病例）、`oa_patient_access_log`（访问审计日志）

**安全特性**：
- 身份证号、手机号自动脱敏展示（前6后4/前3后4）
- 患者信息访问全链路审计日志
- 基于角色的数据权限控制

## 六、系统安全设计

### 6.1 认证与鉴权

- **JWT 无状态令牌认证**：用户登录后颁发 JWT Token，后续请求携带 Token 进行身份验证
- **网关统一鉴权**：Spring Cloud Gateway 拦截所有请求，校验令牌有效性
- **RBAC 权限模型**：基于角色的访问控制，通过 `用户 → 角色 → 菜单/按钮权限` 三级关联实现细粒度权限管理
- **数据权限控制**：`@DataScope` 注解 + AOP 切面，支持全部数据、自定义部门、本部门、本部门及以下等数据范围

### 6.2 安全防护

- **密码安全**：前端 RSA 加密传输（JSEncrypt），后端 BCrypt 哈希存储
- **图形验证码**：登录时 Kaptcha 算术验证码防机器人
- **XSS 防护**：网关层 XSS 过滤器
- **敏感数据脱敏**：身份证号、手机号自动脱敏显示
- **操作审计**：`@Log` 注解自动记录增删改操作日志
- **患者隐私保护**：患者信息访问全链路审计，记录访问人、时间、IP 等

### 6.3 流量治理

- **Sentinel 流控**：各微服务配置独立限流规则（认证服务 500 QPS、系统服务 1000 QPS、代码生成 200 QPS、定时任务 300 QPS）
- **Feign 熔断**：Sentinel 整合 OpenFeign 实现服务调用熔断降级

## 八、关键技术实现

### 8.1 微服务通信

采用 Spring Cloud OpenFeign 声明式 HTTP 客户端实现微服务间调用，`hospital-api-system` 模块定义统一的 Feign 接口，由调用方引入依赖即可透明调用。

### 8.2 统一配置管理

所有微服务配置统一托管至 Nacos 配置中心，支持配置热更新：
- `application-dev.yml`：公共配置（Feign/监控端点）
- `hospital-*-dev.yml`：各服务私有配置（数据源/MyBatis/接口文档等）

### 8.3 文件存储策略

文件服务支持三种存储后端，可按需切换：
- **本地存储**：开发环境使用，文件存储在服务器本地磁盘
- **FastDFS**：分布式文件系统，适合中等规模部署
- **MinIO**：S3 兼容的对象存储，适合云原生部署

### 8.4 数据权限隔离

通过自定义 `@DataScope` 注解 + MyBatis 拦截器，在 SQL 查询时自动注入数据权限条件，实现不同角色只能看到权限范围内的数据，无需手动编写权限过滤逻辑。

### 8.5 前端路由与权限

前端根据用户角色动态生成路由表，菜单权限和按钮权限均由后端控制：
- 登录后通过 `/getRouters` 接口获取当前用户可访问的菜单树
- Vuex 状态管理中动态注册路由
- 按钮级权限通过自定义指令 `v-hasPermi` 控制

## 九、项目特色与创新点

1. **医院行业定制化**：针对医院行政办公场景设计，涵盖患者服务、科室管理、审批流转等医院特有业务需求，而非通用 OA 的简单套用。

2. **患者隐私合规**：患者信息查阅全链路审计，身份信息自动脱敏，符合医疗信息安全规范要求。

3. **微服务弹性架构**：各业务模块独立部署，可根据医院实际负载弹性扩缩容，如患者服务高峰期可单独扩展。

4. **多存储后端适配**：文件服务支持本地/FastDFS/MinIO 三种模式，适应从单机部署到分布式集群的不同场景。

5. **精细化权限体系**：RBAC + 数据权限双重控制，支持按科室、按角色的多维数据隔离，满足医院多科室、多层级的管理需求。

6. **一站式开发工具链**：代码生成器支持从数据库表直接生成前后端代码骨架，提升开发效率。
